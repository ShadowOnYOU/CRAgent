## 1. L1：动态上下文工程层（Context Engine）——最小实现已落地（优先级：高）

手册目标：构建“刚好够用”的上下文（依赖分析 / RAG / 筛选策略 / Checklist），不要把所有信息塞给 LLM。

现状：
- 已新增 `context_engine/`（L1 最小版）：目前仅实现 Checklist 注入。
  - 入口实现：[agent-system/context_engine/checklist.py](agent-system/context_engine/checklist.py)
  - 默认配置文件：[agent-system/config/checklist.json](agent-system/config/checklist.json)
  - 主流程注入点：`CodeReviewAgent.review_pr()`（推理前将 checklist 写入 `Context.checklist`）
- 推理阶段仍以 diff 摘要/原始 diff 为主，上下文构建尚未覆盖“相关文件/函数裁剪”“依赖分析/调用链”“RAG”。

缺口清单（建议分阶段实现）：
- [x] `context_engine/` 目录（最小实现）
- [x] Checklist 注入（业务/工程规则）
- [ ] 基于 diff 的“相关文件/函数”裁剪（最小可用版，不需要向量库）
- [ ] 依赖分析/调用链追踪（Python 可先用 ASTParser 做近似 caller/callee）
- [ ] RAG（向量库 + embedding）
  - 现有配置仍为占位/未接入主流程：[agent-system/config/settings.py](agent-system/config/settings.py)

建议落地顺序：先做“相关文件/函数裁剪 + token 预算”，再做“调用链”，最后再接 RAG。

## 3. LongCoT 的 VERIFY 证据链——已增强但仍不完整（优先级：高）

手册目标：结论必须“带证据”，且证据应能定位到真实代码路径。

现状（已实现/已补齐）：
- [x] 关键词提取升级为 LLM 主导，并带缓存/清洗/回退策略
- [x] 证据收集不再“命中即停”：跨关键词累计证据，带去重与上限控制
- [x] 证据结构化：搜索证据至少包含 file/line/content；并可附加 scope（函数级 snippet/范围）
- [x] 基于证据的判定：对每个假设输出 confirmed/rejected/inconclusive，并把理由/证据带入 CONCLUDE
- [x] 失败处理：证据不足时保持 inconclusive，不把“猜测”当成结论输出

仍然缺口（建议继续补强）：
- [ ] 更系统的“证据链编排”（从 search 到可解释验证）：
  - 先定位符号定义（find_function / ast_analysis）
  - 再找引用（find_references）
  - 再读取局部上下文（read_file + range）
  - 再做基于证据的判断（现已实现，但上游证据质量仍可提升）
- [ ] 证据真实性校验（与 L3 事实校验呼应）：scope/snippet 需能在目标文件中复现命中行
- [ ] 更强的关键词迭代策略：当 inconclusive 时，利用 next_keywords 做多轮收敛（目前为轻量合并缓存）

相关代码入口：
- 推理引擎：[agent-system/reasoning/long_cot.py](agent-system/reasoning/long_cot.py)
- 工具集成：[agent-system/agents/tool_agent.py](agent-system/agents/tool_agent.py)

## 4. L3：结果过滤层（LLM-as-Judge）——事实校验已补齐（优先级：最高）

手册目标：过滤误报/低价值/重复，并进行事实校验（防幻觉）。

现状：
- 已有：严重级别/置信度阈值、低价值关键词过滤、简单重复判定、可选 LLM Judge。
- 已补齐：事实校验与强约束（strict_facts=True 时生效；主流程默认启用）。

已实现清单：
- [x] issue 的 file/line 校验：
  - file_path 必须是安全的相对路径（拒绝绝对路径/.. 上跳）
  - line_number 必须 >0 且能定位
  - 优先用 root_path 下的真实文件做行号范围校验；读不到文件时退化到 diff hunk 范围校验
- [x] evidence 校验：证据片段必须能在目标文件/PR diff/new_content 中复现（至少字符串匹配），否则降级置信度
- [x] 强制结构：无证据直接降为 0 置信度（通常会被阈值过滤掉）

仍然缺口（建议继续补强）：
- [ ] 更精确的 diff 事实源：多 hunk 的 new_line → diff_line 映射（用于更严格的“定位到具体 hunk/行”）
- [ ] 结果结构更强约束：将 evidence 从“字符串列表”升级为结构化（file/line/range/snippet/hash），便于复盘与强校验
- [ ] 对 LLM Judge 的输出做更严格校验（keep/remove 是否越界、是否产生不可定位 issue 等）

相关代码入口：
- 过滤器：[agent-system/judge/filter.py](agent-system/judge/filter.py)
- PR diff 解析（用于构建“变更范围”事实源）：[agent-system/ingestion/pr_parser.py](agent-system/ingestion/pr_parser.py)

## 5. PR 解析模块：行号/多 hunk 支持不完善（优先级：中）

现状：
- `parse_diff` 能按 `diff --git` 切文件，但行号范围计算非常粗：只取第一个 hunk 的 `+start`，再用“新增行数”估算 line_end。
- 对多 hunk、rename、binary diff、更复杂格式支持不足。

缺口清单：
- [ ] 支持多 hunk（每个 hunk 的 new_start/new_count）
- [ ] 更准确的“新增/删除行”映射（生成 new_line → diff_line 的映射表）
- [ ] rename/binary 等 diff 情况的健壮处理

相关代码入口：[agent-system/ingestion/pr_parser.py](agent-system/ingestion/pr_parser.py)
