# 项目运行流程详解（以 sample-lib 示例 diff 为例）

本文档面向“读代码/调试/二次开发”，解释当前仓库在一次完整评审中，如何从 diff 输入一步步走到最终结论输出，并重点说明 LongCoT 的 VERIFY 阶段是如何做“递归挖掘式检索”的（含 token/耗时上限控制）。

示例输入：
- diff 文件：`sample-lib/pr_diffs/001-introduce-cache-bug.diff`
- 代码根目录：`sample-lib/`

---

## 0. 先跑起来（命令行）

### 0.1 必要环境变量

主入口会强制检查 `DASHSCOPE_API_KEY`：没设置会直接退出。

```bash
export DASHSCOPE_API_KEY=your_api_key_here
```

### 0.2 macOS/conda 下推荐的 python 调用方式

在你的机器上 `python` 可能仍会被系统 Python 2.7 抢占，因此推荐显式指定解释器（和你 Debug 命令同一思路）：

```bash
cd agent-system
/usr/bin/env /opt/anaconda3/envs/cragent/bin/python main.py \
  --diff ../sample-lib/pr_diffs/001-introduce-cache-bug.diff \
  --root ../sample-lib \
  --title "SampleLib: Introduce cache bug" \
  --output ../sample-lib/review_output.json
```

## 1. 总入口：CLI → Agent 组装

入口文件：`agent-system/main.py`

### 1.1 CLI 参数解析

`main()` 支持：
- `--diff/-d`：diff 文件路径
- `--root/-r`：代码根目录（工具检索的 root）
- `--title/-t`：PR 标题
- `--simple/-s`：简单模式（不走 LongCoT）
- `--no-filter`：禁用过滤
- `--output/-o`：把结果 JSON 额外写到指定路径

### 1.2 环境检查

- 若 `DASHSCOPE_API_KEY` 未设置：打印提示并 `sys.exit(1)`。

### 1.3 CodeReviewAgent 初始化

`CodeReviewAgent.__init__(root_path, use_long_cot)` 会组装核心组件：
- `LLMClient()`：负责调用百炼 OpenAI 兼容接口（并把 llm_request/llm_response 记入 run_trace）
- `ToolAgent(root_path)`：工具层，提供 `code_search/find_function/find_references/read_file/ast_analysis/grep/get_function_context`
- `PRParser()`：diff 解析
- `ReviewFilter(LLMClient)`：过滤器（阈值/低价值/去重 + 可选 LLM Judge）
- `FeedbackLoop()`：结果与反馈落盘
- `LongCoTEngine(LLMClient, ToolAgent)`：默认推理引擎（READ→HYPOTHESIZE→VERIFY→CONCLUDE）

---

## 2. 输入解析：diff → PR/CodeChange

入口：`CodeReviewAgent.review_diff(diff_text, pr_title)`

### 2.1 parse_diff

`PRParser.parse_diff(diff_text)` 会把 diff 切成若干 `CodeChange`：
- `file_path`
- `diff`（unified diff 文本）
- `line_start/line_end`（当前实现为粗略范围，用于展示/后续可改进）

### 2.2 组装 PR

`review_diff` 会创建一个 `PR`：
- `pr_id`：基于 diff hash
- `title/description`
- `changes`：解析得到的 `CodeChange[]`

之后进入：`review_pr(pr)`。

---

## 3. 推理主线（LongCoT）：READ → HYPOTHESIZE → VERIFY → CONCLUDE

入口：`LongCoTEngine.reason(pr, context=None)`（文件：`agent-system/reasoning/long_cot.py`）

### 3.1 run_trace：全链路可复盘

在 `reason()` 开头会写入：
- `run_start`：包含 engine/model/pr_id/title/changed_files

后续每次：
- LLM 调用会写入 `llm_request/llm_response/llm_error`（在 `LLMClient.chat()` 内）
- 工具调用会写入 `tool_request/tool_response`（在 LongCoT 内部显式 append）

最后写入：
- `run_end`：issues 数量与 summary

这些会进入：
- 结果对象的 `result.run_trace`
- `--output` 指定的 JSON 文件
- `./data/feedback/result_<pr_id>.json`（`FeedbackLoop.save_review_result`）

---

## 4. READ：构建“可读且可控预算”的输入

函数：`LongCoTEngine._read(pr, context, run_trace)`

关键点：
1) 先调用 `_build_code_summary(pr, context)`：
   - 必须包含真实 diff（每个文件一个 diff block），否则无法做代码级理解
   - 按 `config.review.max_context_tokens` 近似换算成字符预算，并做“中间截断”（保留头尾）
   - 若有 `Context`（目前多为占位/外部传入），会把 `related_files/checklist/docs_context/code_context` 拼入输入

2) 再用 `READ_SYSTEM` 提示词调用 LLM：
   - 这一步主要做“初读”：识别关键点、并发/资源/边界等风险方向

输出（保存到 read_result）：
- `raw_response`（LLM 的原文）
- `code_summary`（喂给 LLM 的摘要/截断后的 diff）

---

## 5. HYPOTHESIZE：产出可验证的缺陷假设（JSON + fallback）

函数：`LongCoTEngine._hypothesize(pr, read_result, run_trace)`

流程：
1) 强制要求 LLM 输出 JSON：
   ```json
   {"hypotheses": ["...", "..."]}
   ```
2) 解析策略：
   - 优先 `LLMClient.extract_json()`
   - 若失败，走 `_extract_hypotheses_fallback()` 兼容 Markdown/“假设：...”格式
3) 去重 + 限制最多 5 条

产出：`hypotheses: List[str]`

---

## 6. VERIFY：递归挖掘式检索（带上限）

这是你提出要“细化”的部分：现实 code review 往往需要沿调用链/引用链继续向下挖，直到证据足够。

当前实现已经把 VERIFY 从“一次关键词命中”升级为“可控的递归挖掘”，入口：
- `LongCoTEngine._verify()`
- 内部调用 `LongCoTEngine._mine_evidence_until_conclusion()`

### 6.1 第一步：生成 seed keywords（LLM 优先）

函数：`_extract_search_keywords(hypothesis, run_trace)`

- LLM 根据假设生成 `keywords`（带缓存/清洗/回退）
- 同时会用正则提取“保底锚点”（标识符/点号链/引号片段/中文短语）
- 合并去重后最多 8 个

### 6.2 第二步：进入递归深挖循环（BFS）

函数：`_mine_evidence_until_conclusion(pr, hypothesis, seed_keywords, run_trace)`

它把“挖掘任务”放到队列里（BFS），避免一条路径挖太深。

#### 6.2.1 预算上限（防 token/耗时爆炸）

当前默认硬编码上限（后续可以迁移到 config）：
- `max_depth = 2`：最大挖掘深度
- `max_tool_calls = 14`：最多工具调用次数（覆盖 code_search/find_function/find_references/get_function_context 等）
- `max_evidence_items = 15`：最多证据条数
- `max_scopes_to_attach = 5`：最多给证据附加函数级 scope 的条数
- `max_symbols_to_follow = 8`：每轮最多跟进的候选符号数量

> 这些上限共同决定“最多挖多深、挖多久、喂给 LLM 的证据有多少”。

#### 6.2.2 任务类型

队列里的 task 有两类：
- `keyword`：触发 `code_search(pattern=keyword)`
- `symbol`：触发 `find_function(function_name=symbol)` + `find_references(symbol=symbol)`

#### 6.2.3 证据采集与去重

所有工具返回的命中会被结构化成：
- `file/line/content`

并使用 `(file,line,content)` 做去重，直到达到 `max_evidence_items`。

#### 6.2.4 “函数级证据”：get_function_context

为了避免只有“一行命中”过粗，系统会对部分证据补充 `scope`：
- Python 文件优先用 AST 找到命中行所在最小函数范围
- 返回：`scope_type/name/line_start/line_end/snippet/truncated`

这使得后续 VERIFY_EVAL/CONCLUDE 有更可靠的上下文片段可引用。

#### 6.2.5 从证据中抽取符号，继续向下挖

系统会从 `content` 和 `scope.snippet` 里抽取候选符号：
- `def foo(` / `class Bar`
- `obj.method` / `module.func`
- `foo(` 调用形态（过滤 if/for/return 等噪声）

对抽取到的符号，会生成 `symbol` 任务，从而把检索从“关键词命中”推进到“跟定义/跟引用”。

#### 6.2.6 每挖一段就评估一次（是否可以下结论）

当证据累积到一定量（例如 ≥3 条，或深度 ≥1），会调用：
- `_evaluate_hypothesis_with_evidence(hypothesis, evidence, run_trace)`

LLM 输出：
```json
{
  "status": "confirmed|rejected|inconclusive",
  "confidence": 0.0,
  "reason": "...",
  "next_keywords": ["...", "..."]
}
```

- 若 `confirmed/rejected`：立即停止深挖
- 若 `inconclusive`：把 `next_keywords` 追加进队列（作为下一层 keyword 任务）

这就是“递归挖掘直到结论或上限”的闭环。

### 6.3 VERIFY 的多轮迭代

`reason()` 顶层还有 `max_iterations`（默认 3）外循环：
- 若某些假设仍 `inconclusive`，会带着更新过的关键词缓存进入下一轮
- 每轮都会做一次新的深挖尝试，但仍受上限约束

---

## 7. CONCLUDE：只基于证据输出结果

函数：`LongCoTEngine._conclude(pr, verification_results, run_trace)`

它会把每个 confirmed 的假设：
- `hypothesis`
- `evaluation.reason`
- `evidence`（含 scope snippet）

拼成 `evidence_summary`，再交给 LLM 输出最终 ReviewResult JSON。

---

## 8. 过滤与落盘

### 8.1 过滤（L3）

入口：`CodeReviewAgent.review_pr(... enable_filter=True)`

默认过滤器：`ReviewFilter.filter()`
- 严重级别阈值（默认 low）
- 置信度阈值（默认 0.5）
- 低价值文本过滤（如“建议加注释/命名不规范”等）
- 简单去重（同 file + issue_type）

> 注意：目前过滤器仍缺“事实校验”（例如 file/line 是否真实存在）——这是后续优先补齐方向。

### 8.2 落盘

1) 总是会写入：`./data/feedback/result_<pr_id>.json`
   - 包含 summary/issues/reasoning_trace/run_trace/timestamp

2) 若 CLI 传了 `--output`：还会额外写一份 JSON（便于调试/对比）。

---

## 9. 用本仓库示例理解“递归挖掘”会怎么工作

示例 diff：`sample-lib/pr_diffs/001-introduce-cache-bug.diff`

这个例子里，关键问题在于：`LRUCache.get_or_set()` 的并发语义（factory 在锁外导致重复执行/覆盖写）。

一次典型的深挖路径可能是：
1) seed keywords 命中 `get_or_set` / `LRUCache` / `factory` / `lock`
2) `code_search` 命中 `samplelib/lru.py` 的实现
3) `get_function_context` 把证据提升到函数片段（包含 get→factory→set 的完整逻辑）
4) 从 snippet 抽取符号（如 `get_or_set`, `factory`, `set`）并尝试：
   - `find_function(get_or_set)` 定位定义
   - `find_references(get_or_set)` 查看调用方（是否存在并发调用场景、是否要求单次执行）
5) 证据足够后，VERIFY_EVAL 输出 `confirmed`，CONCLUDE 给出带证据的评审结论

## 10. 常见调试点

1) 想看每一步做了什么：
- 关注输出 JSON 里的 `reasoning_trace`（人类可读）
- 关注 `run_trace`（机器可复盘，含每次 LLM/tool 的输入输出截断）

2) 想控制“挖掘上限”：
- 目前在 `LongCoTEngine._mine_evidence_until_conclusion()` 内部硬编码了 `max_depth/max_tool_calls/...`
- 后续建议迁移到 `config.review`，并在 CLI 提供覆盖参数

3) 想看工具返回的原始命中：
- 搜索 `run_trace` 中 `type=tool_response` 的 `result`
